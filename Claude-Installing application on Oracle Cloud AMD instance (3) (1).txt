Your Proposed Stack — Tool-by-Tool Assessment


SearXNG — correct tool, wrong primary role


SearXNG is a self-hosted meta-search engine that aggregates results from Google, Bing, DuckDuckGo, and others without API keys or rate limits. Your instinct to include it is right, but it should be the fallback layer, not the primary one. JobOps already has purpose-built extractors (JobSpy, Adzuna, startup.jobs) that return structured data — title, employer, salary, URL — with no HTML parsing needed. SearXNG returns unstructured search snippets. Use it specifically for finding direct career page URLs that the structured extractors miss, and for smaller employers who only post on their own website.

Firecrawl — replace it entirely


Firecrawl is a managed cloud service. You pay per crawl, it sends your data to their servers, and you have no control over the browser fingerprint it presents. For job applications specifically this matters — ATS vendors detect and block cloud scraping infrastructure by ASN. SCRAPLING is the correct replacement. It is a pure Python library, runs entirely locally, handles both static and JavaScript-rendered pages, and has built-in support for Playwright and Camoufox as rendering backends. The API is also significantly cleaner than Firecrawl's.


# SCRAPLING handles both cases cleanly
from scrapling import Fetcher, PlaywrightFetcher

# Static pages (most job boards' API endpoints)
fetcher = Fetcher(auto_match=True)
page = fetcher.get('https://boards.greenhouse.io/company/jobs/12345')

# JavaScript-rendered pages (Workday, custom ATS)
fetcher = PlaywrightFetcher(browser_type='camoufox')
page = fetcher.get('https://company.wd5.myworkdayjobs.com/careers')


Camoufox — keep it, it's the right choice


Camoufox is already embedded in JobOps (the UKVisaJobs and Hiring Cafe extractors use it). It patches Firefox at the C++ level to randomize canvas fingerprints, WebGL renderer strings, font metrics, audio context noise, and hardware concurrency values. This is meaningfully harder to detect than standard Playwright with stealth plugins. Your instinct here is correct. The key constraint is memory — a single Camoufox instance uses 300–500 MB. On your 1 GB Oracle instance, you can run at most one browser instance at a time, which means all browser-dependent operations must be serialized, never parallel.


The gap in your stack — ATS drivers


This is the part your proposal doesn't address and it's where most auto-application projects fail. Each major ATS has a completely different DOM structure, form field naming convention, and submission flow. You cannot write one generic "fill this form" agent and have it work across all of them. You need separate drivers per ATS:


Greenhouse   — React SPA, fields use data-qa attributes, uploads go to S3 presigned URLs
Lever        — Similar React structure but different field IDs, often has Typeform-style screens
Workday      — The worst one. Angular app, shadow DOM elements, fields change IDs on every page load
Taleo        — Old Oracle system, mostly server-rendered, but multi-step with session cookies
LinkedIn Easy Apply — Their own API, can be called directly if you have a session token
Indeed Apply — Also has a direct API path for one-click applications
Custom ATS   — No pattern, pure Camoufox required, highest failure rate


The practical approach for a first version is to implement only Greenhouse and Lever. Together they cover roughly 60% of tech/startup job postings, they have relatively stable DOM structures, and they don't use Workday's shadow DOM hell.

# apply-agent/main.py — minimal FastAPI skeleton
from fastapi import FastAPI
from pydantic import BaseModel
from ats_detector import detect_ats
from drivers import greenhouse, lever, linkedin_easy

app = FastAPI()

class ApplicationRequest(BaseModel):
    job_id: str
    application_url: str
    pdf_base64: str           # tailored resume PDF
    cover_letter: str         # generated by Ghostwriter
    candidate_profile: dict   # name, email, phone, address, linkedin
    screening_answers: dict   # question -> answer, pre-generated by AI

class ApplicationResult(BaseModel):
    success: bool
    confirmation_id: str | None
    error: str | None
    ats_type: str
    screenshot_base64: str | None  # confirmation page screenshot

@app.post("/apply", response_model=ApplicationResult)
async def apply(req: ApplicationRequest):
    ats_type = await detect_ats(req.application_url)
    
    driver_map = {
        "greenhouse": greenhouse.apply,
        "lever": lever.apply,
        "linkedin_easy_apply": linkedin_easy.apply,
    }
    
    driver = driver_map.get(ats_type)
    if not driver:
        return ApplicationResult(
            success=False,
            error=f"No driver for ATS type: {ats_type}",
            ats_type=ats_type,
            confirmation_id=None,
            screenshot_base64=None
        )
    
    return await driver(req)


ATS detection is the first thing you build because everything else depends on knowing which driver to invoke:


# apply-agent/ats_detector.py
from scrapling import Fetcher
import re

ATS_PATTERNS = {
    "greenhouse":        r"boards\.greenhouse\.io|greenhouse\.io/embed",
    "lever":             r"jobs\.lever\.co|hire\.lever\.co",
    "workday":           r"myworkdayjobs\.com|workday\.com/.*jobs",
    "taleo":             r"taleo\.net|tbe\.taleo\.net",
    "linkedin_easy_apply": r"linkedin\.com/jobs/view/.*easyApply",
    "indeed_apply":      r"indeed\.com/.*indeedapply",
    "smartrecruiters":   r"careers\.smartrecruiters\.com",
    "ashby":             r"jobs\.ashbyhq\.com",
}

async def detect_ats(url: str) -> str:
    # First pass: URL pattern matching (fast, no HTTP request)
    for ats, pattern in ATS_PATTERNS.items():
        if re.search(pattern, url, re.IGNORECASE):
            return ats
    
    # Second pass: fetch and inspect the page HTML
    try:
        fetcher = Fetcher(auto_match=False)
        page = fetcher.get(url, timeout=10)
        html = page.content.lower()
        
        if "greenhouse" in html or "gh_jid" in html:
            return "greenhouse"
        if "lever.co" in html or "lever-job-listing" in html:
            return "lever"
        if "workday" in html or "wd5.myworkdayjobs" in html:
            return "workday"
        if "taleo" in html:
            return "taleo"
        if "smartrecruiters" in html:
            return "smartrecruiters"
        if "ashbyhq" in html:
            return "ashby"
    except Exception:
        pass
    
    return "unknown"

async def _fill_if_present(page, selector, value):
    try:
        element = page.locator(selector)
        if await element.count() > 0:
            await element.fill(value)
    except Exception:
        pass  # Field not present on this specific posting

async def _fill_screening_questions(page, answers: dict):
    # Greenhouse renders custom questions with varying field types
    questions = await page.locator('[data-qa^="question_"]').all()

One Critical Recommendation: Build This as a Copilot, Not an Autopilot


Fully autonomous job applications carry real risks. A bug in the salary question parser submits "I expect $0" to 30 companies before you notice. An ATS fingerprinting system bans your IP and your accounts. A poorly generated answer to "why do you want to work here" goes to a company where you know the hiring manager.


The architecture that actually works in practice has a human approval gate between Layer 3 and Layer 4 in the diagram above. The agent does all the hard work — detects the ATS, fills every field, generates every answer, uploads the PDF — then pauses and shows you a preview. You see a rendered summary of exactly what will be submitted. One click to approve, one click to skip. This is how most production tools in this space (Simplify, LazyApply, Sonara) actually work, and why they have better reputations than the fully-autonomous ones.


In JobOps terms, this would surface as a new panel on the Ready tab — "Pending approval: 3 applications ready to submit" — showing the generated answers for each, with Approve and Skip buttons. Only after approval does the apply-agent service receive the payload and execute.

Question 1: Can Scrapling and Camoufox Handle Form Execution?


No — and this is a critical distinction that most people miss.


Scrapling solves a fundamentally different problem than form filling. Understanding why requires separating two operations that look similar but are architecturally opposite:

Operation
Direction
Scrapling relevance
Scraping
Read DOM → extract data
Core use case. Auto-match finds elements even when selectors change
Form filling
Write data → interact with DOM
Not what Scrapling is for

Scrapling's auto-match system works by fingerprinting an element across multiple signals simultaneously — CSS selector, XPath, text content, relative position to siblings, element attributes — and rebuilding a match when any one signal breaks. This is genuinely powerful for finding where a piece of data lives on a page.


But form filling isn't about finding an element once. It's about driving a stateful interaction sequence — click field, type text, handle the framework's synthetic events, move to next step, detect validation errors, handle conditional fields that appear based on previous answers, upload a file through a dialog, hit submit, verify the confirmation page loaded. Each of those steps can fail independently. The brittleness problem for form filling is not "I can't find the email field" — it's "I found the email field but React rejected my value because I didn't trigger the right synthetic event" or "step 3 of 4 in Workday now has a new mandatory question I didn't account for."


Camoufox solves the detection problem, not the navigation problem. It makes your browser session look human. It doesn't make your interaction logic resilient to DOM changes.



The Tools That Actually Solve Interaction Brittleness


The correct framing is: instead of hardcoding selectors, let an LLM look at the page and decide what to interact with. Two open-source projects do this properly:


Browser Use (browser-use/browser-use on GitHub) is the most directly relevant. It extracts all interactive elements from the current page DOM, serialises them into a compact representation, passes that to an LLM with a task description, and the LLM returns which element to interact with and how. When a site redesigns and changes all its class names, the LLM still identifies "this is the email input field" from semantic context rather than a fragile selector. It uses Playwright as the underlying browser driver and supports Claude and GPT-4 as the decision model.


from browser_use import Agent
from langchain_anthropic import ChatAnthropic

agent = Agent(
    task="Fill in the job application form at this URL with these details: "
         "name John Smith, email john@example.com, upload resume.pdf, "
         "answer 'Why do you want to work here?' with this text: [generated answer]",
    llm=ChatAnthropic(model="claude-sonnet-4-6"),
)
await agent.run()


That single task description handles element identification, multi-step navigation, and conditional fields — without a single hardcoded selector.


Skyvern (Skyvern-AI/skyvern on GitHub) takes this further. It was built specifically for "automate browser workflows that are resilient to UI changes," which is precisely your use case. It combines computer vision (screenshots) with DOM analysis and passes both to an LLM. The visual component handles cases where the DOM is unhelpful — shadow DOM elements in Workday, iframes, elements that only exist as painted pixels. Skyvern has direct first-class support for file uploads, multi-step form sequences, and has built-in 2captcha integration for CAPTCHA challenges. It also ships with a self-hosted Docker image.

Do You Still Need ATS-Specific Knowledge?


Yes, but at a much higher level. The thing that remains ATS-specific is not selector paths but workflow knowledge:

The term you're half-remembering is content scripts — the mechanism by which browser extensions inject JavaScript that runs directly inside a webpage's context.


Here's the fundamental architectural difference that makes this approach categorically different from server-side Playwright:


Server-side Playwright (what we discussed before):

The extension is you. It runs in your browser, in your session, with your IP address, with your cookies, with your accumulated browsing history that feeds reCAPTCHA v3's behavioral scoring. How Simplify Actually Works


Simplify is a Chrome extension. When you're on a job application page, here is the precise sequence:


Step 1 — Detection. The extension's background service worker checks every tab URL against a list of known ATS patterns (same boards.greenhouse.io, jobs.lever.co patterns we discussed). When a match is found, it activates the content script for that tab.


Step 2 — DOM injection. The content script runs inside the job application page. It scans the DOM for form fields and injects small UI elements (the Simplify icons) adjacent to each field. This is possible because content scripts have full access to document.querySelector, document.createElement, and every other DOM API.


Step 3 — Profile data. When you click a Simplify icon next to a field, the content script sends a message to Simplify's background service worker, which calls Simplify's backend API. The API returns the right value from your stored profile — email, phone, address, years of experience, etc.


Step 4 — Form filling. The content script writes the value back into the field. This is where it gets technically subtle. You cannot just do field.value = 'text' on a React or Angular form. Modern JS frameworks intercept input through synthetic events, not by watching .value. The correct technique is:


// Naive approach — breaks React/Angular/Vue
field.value = 'john@example.com';

// Correct approach — triggers framework's synthetic event system
const nativeInputValueSetter = Object.getOwnPropertyDescriptor(
    window.HTMLInputElement.prototype, 'value'
).set;
nativeInputValueSetter.call(field, 'john@example.com');
field.dispatchEvent(new Event('input', { bubbles: true }));
field.dispatchEvent(new Event('change', { bubbles: true }));


This is why naive autofill extensions break on React-based ATS platforms. Simplify handles this because they've built the correct synthetic event dispatch for each major framework.


Step 5 — AI-generated answers. For open-text screening questions like "Why do you want to work here?", Simplify's content script captures the question text, sends it along with the job description to Simplify's backend, which calls an LLM API (they use OpenAI), gets the generated answer back, and fills it into the textarea using the same event dispatch technique.


Step 6 — Reporting. After submission, the content script detects the confirmation page (usually a URL change or a DOM element with confirmation text), extracts the confirmation number if present, and syncs back to Simplify's backend to mark the application as submitted.
